# 会话恢复

cosh-shell 可以发现并恢复当前工作空间中过往的 Agent 对话。恢复会重新载入模型
可见的消息，使下一次请求延续原对话，而不是开启全新轮次。

会话恢复适用于 `cosh-core` adapter。它不会恢复历史终端输出、审批提示、用户
问题或其他临时 UI 状态。`/session status` 会明确说明这一边界，避免把旧终端
证据误呈现为当前证据。

## 从历史会话启动

Shell 就绪后打开选择器：

```bash
cosh-shell --resume
cosh-shell raw cosh-core --resume
```

直接选择已知的规范会话 UUID：

```bash
cosh-shell --resume 2d711642-b726-4b04-8d2a-8a0470f4ed24
cosh-shell raw cosh-core --resume 2d711642-b726-4b04-8d2a-8a0470f4ed24
```

`--resume` 及其可选值属于启动参数，不会被误识别为 adapter 名。无论通过启动
参数、交互式选择器，还是直接使用 `cosh-core --resume`，都会进入相同的验证和
持久化路径。

## 在 Shell 中管理会话

可在 Shell 提示符中使用：

| 命令 | 行为 |
|------|------|
| `/session` | 打开按更新时间倒序排列的会话选择器 |
| `/session list` | 不打开选择器，直接打印首个有界摘要分页 |
| `/session status` | 显示 Shell、已选择、恢复中和已激活的 provider 身份 |
| `/session resume <id>` | 验证并选择一个 provider 会话 |
| `/resume [id]` | `/session` 或 `/session resume <id>` 的别名 |
| `/session clear <id>...` | 清理指定 ID 前请求确认 |
| `/session clear --all` | 准备精确 ID，并在清理全部持久化会话前请求确认 |

选择会话本身不会调用模型。下一次 Agent 请求发出时，状态从 `selected`
变为 `restoring`；只有恢复后的请求成功完成，状态才变为 `active`。恢复失败是
可处理的，Shell 提示符仍然可用。直接恢复和选择器恢复都会在 Agent 运行或其他
交互决策处于活动状态时拒绝变更选择。

active 会话自动续接时，如果持久化记录缺失、损坏、版本不兼容或作用域不一致，
cosh-shell 会释放失效 active ID，而不会永久重复恢复。认证、预算、模型等普通
provider 错误仍保留 active 会话，便于稍后重试。内部单轮 fallback 禁用
provider resume 时，也不会消费用户显式选择但尚未恢复的会话。

## 选择器按键

| 按键 | 操作 |
|------|------|
| `Up` / `Down`、`j` / `k` | 移动光标 |
| `Enter` | 恢复高亮的健康会话 |
| `Space` | 标记或取消标记待清理条目 |
| `d` | 对已标记条目或当前高亮条目打开清理确认 |
| `y` | 确认精确的清理集合 |
| `n`、`Esc`、`Ctrl-C` | 取消确认或关闭选择器 |

每一行会显示首条提示摘要或会话 ID、相对更新时间、消息数、模型、健康状态和
保护状态。只有当导航接近已加载边缘时，选择器才会读取下一个有界 cursor 分页，
并仅渲染当前条目附近的有限视窗。Core 会在传输前规整并限制首条提示，终端行
还会进一步截断。

## 工作空间作用域与存储

会话归属于规范化后的当前工作空间。即使把其他工作空间的文件复制进当前目录，
也无法误恢复该会话。

默认持久化根目录为：

```text
~/.copilot-shell/cosh-core/sessions/
```

Core 为每个工作空间使用确定性的哈希子目录，并将每段对话保存为版本化 JSON
信封。信封包含规范 provider UUID、工作空间、时间戳、模型、并发代数和模型
可见消息。写入过程使用同级临时文件、文件系统同步、原子重命名和乐观代数检查。
列表会先对文件元数据排序，并只读取当前请求分页。单个会话文件限制为 32 MiB；
超限条目无需加载其内容即可显示为 `corrupt`。

只有能够证明旧版扁平目录归请求工作空间所有时，已知 UUID 才可从旧布局恢复。
对原先的相对默认值，Core 检查 `<workspace>/sessions/<uuid>.json`；不会认领
作用域不明确的共享根目录或启动器 cwd 下的文件。显式查找只在内存中加载旧数组，
不改写它；只有恢复轮次成功持久化后才创建 schema v1 副本并删除旧文件。能够
证明归当前工作空间所有的旧版会话会与已升级条目一同显示在选择器和
`/session list` 中；它们的 ID（包括损坏文件）也会纳入全部清理，并可通过
显式清理删除。在已确立目录之外的来源不明文件永远不会被列出或认领。

在 Unix 上，工作空间目录会明确限制为 `0700`，会话 JSON、临时文件和锁文件
使用 `0600`，不依赖进程 umask。并发写入使用 advisory 文件锁；进程异常退出
时，内核会自动释放该锁。

可通过用户配置或 `<workspace>/.copilot-shell/config.toml` 中的
`session.persist_dir` 修改根目录。即使 Core 从其他 cwd 启动，项目配置和
相对路径也会从 cosh-shell 发送的工作空间解析。设置
`session.auto_persist = false` 后，轮次历史只保留在当前 Core 进程内。Core
会标记该 ID 不可恢复；即使本轮失败、被取消或异常退出，cosh-shell 也不会在
下一轮提交或复用它。只有本次调用实际使用的身份会失效，无关的 selected ID 和
旧 active ID 会继续保留。

## 健康状态与恢复错误

选择器会保留异常条目，便于识别和清理：

| 健康状态或错误 | 含义 | 后续操作 |
|----------------|------|----------|
| `ready` | 信封有效且属于当前工作空间 | 正常恢复 |
| `corrupt` | JSON 或必需信封字段损坏 | 核对 ID 后确认清理 |
| `incompatible` | 当前版本不支持该 schema | 升级 cosh-core 或清理 |
| `scope_mismatch` | 记录的工作空间不一致 | 返回原工作空间 |
| `not_found` | 列出后文件被删除 | 刷新选择器 |
| `conflict` | 其他写入者持有或推进了会话 | 等待完成后重试 |

损坏、缺失、不兼容、作用域不匹配和并发冲突都不会终止交互式 Shell。只有
`ready` 条目可恢复；异常条目仍可在确认后清理。

每次恢复时，cosh-shell 都会在提交本轮前验证 Core 返回的 provider ID 与
selected 或 active ID 完全一致。类型化持久化失败只会释放本次调用使用的
active 身份，既避免重复恢复陈旧历史，也不会影响无关选择。

## 清理保护

清理始终需要显式请求和确认，确认界面会标明要删除的准确 ID 或数量。已选择
会话和当前活跃 provider 会话在 cosh-shell 与 cosh-core 两层都受到保护，因此
即使出现在清理全部请求中也会被跳过。取消确认不会修改任何记录。
如果所有持久化会话均受保护，命令会显示受保护数量，而不会错误提示工作空间
为空。
